## 1. 미션 (Mission)

이 과제는 "칸반 보드를 처음부터 만드는 능력"을 보지 않습니다. **기본 UI는 이미 제공**됩니다.
우리가 보는 것은 다음 두 가지입니다.

1. **현실의 네트워크 조건**(지연·실패·충돌·대량 데이터) 아래에서 사용자 경험을 어떻게 지켜내는가
2. **정답이 없는 상황**에서 어떤 결정을 내리고, 왜 그렇게 했는가

> 완벽한 기능 개수보다, **하나라도 끝까지 안 깨지게** 만드는 사람을 찾습니다.
> "동작하는 일부가 완벽한 전체보다 낫습니다."

## 2. 요구사항

### P1 — 필수 (견고성) ★ 실질 평가가 여기서 이뤄집니다

1. **데이터 로드 & 상태 표현**
   - `GET /api/tasks`로 데이터를 불러와 보드에 렌더링
   - **로딩 중 / 에러(재시도 가능) / 빈 상태**를 각각 명확히 표현
2. **낙관적 업데이트 + 실패 롤백** *(과제의 핵심)*
   - 드래그 이동·수정·삭제·생성 시 **UI를 즉시 반영**
   - 서버 요청이 실패(약 15%)하면 **이전 상태로 정확히 되돌리고** 사용자에게 알림
   - "매번 스피너 돌리고 성공해야 반영"하는 방식은 요구사항 미충족입니다
3. **경쟁 상태(race condition) 처리**
   - 같은 카드를 **빠르게 연속 이동**해도 최종적으로 **UI와 서버 상태가 일치**해야 함
   - 늦게 도착한 오래된 응답이 최신 상태를 덮어쓰지 않도록 처리
4. **대량 데이터 성능 (5,000개)**
   - 5,000개 태스크에서도 검색/필터/드래그가 **버벅이지 않아야** 함
   - 목표 감각: 검색어 입력 시 체감 지연 없이 결과 반영. 방법(가상화·메모이제이션·인덱싱 등)은 자유
5. **태스크 CRUD**
   - 추가(제목·우선순위 필수, 설명 선택), 수정, 삭제(확인 다이얼로그)
6. **핵심 로직 유닛 테스트**
   - 필터/정렬/상태 전이/롤백 등 순수 로직의 정확성을 테스트로 증명 (전체 커버리지가 아니라 핵심만)

### P2 — 권장 (시간이 되면)

1. **409 충돌 처리** — 다른 곳에서 먼저 수정되어 충돌 시의 UX
2. **재시도 / 백오프** — 실패한 쓰기 요청 재시도 전략
3. **다중 탭 동기화** — 한 탭의 변경을 다른 탭에 반영(`BroadcastChannel` 등)
4. **키보드 접근성** — 키보드만으로 카드 이동, ARIA, 포커스 관리
5. **검색/필터 고도화** — 디바운싱, 우선순위·상태·태그 다중 필터(AND)

### 일부러 명세하지 않은 것 (판단력 평가)

아래는 **정답을 주지 않습니다.** 스스로 결정하고 `DECISIONS.md`에 **무엇을 왜 선택했는지** 적으세요.
결정 자체보다 **근거의 타당성**을 봅니다.

- 네트워크가 **완전히 끊겼을 때** 어떻게 동작해야 하는가? (읽기전용 잠금 / 오프라인 큐잉 / 에러 등)
- **409 충돌**이 났을 때 사용자에게 무엇을 보여주고 어떻게 해소하는가?
- 실패한 쓰기를 **자동 재시도**할지, 사용자가 직접 재시도하게 할지

### 평가하지 않는 것 (여기에 시간 쓰지 마세요)

- 다크 모드, 화려한 애니메이션, 정교한 비주얼 디자인 — **채점하지 않습니다.** 견고성에 집중하세요.

## 3. 접근 가이드 (해결 방안 힌트)

정답 코드는 없지만, 막히지 않도록 방향만 제시합니다. **어떻게 풀지는 당신의 선택**입니다.

- **출발점**: `src/Board.tsx`의 `TODO(P1)` 주석부터 보세요. 현재는 서버에 저장하지 않고 로컬 상태만 바꾸는 "순진한" 구현입니다.
- **낙관적 업데이트 & 롤백**:
  - 요청을 보내기 **전에** 화면을 먼저 바꾸고, 실패하면 되돌립니다.
  - 되돌릴 때 "서버를 다시 조회"할 수도, "바꾸기 직전 스냅샷을 보관해뒀다 복원"할 수도 있습니다. 각 방식의 장단점을 생각해 보세요.
  - `src/api/client.ts`의 `ApiError`는 `status`/`payload`를 담고 있어, 409 시 `payload.current`로 서버 최신 상태를 읽을 수 있습니다.
- **경쟁 상태**:
  - 지연이 랜덤이라 **먼저 보낸 요청이 나중에 도착**할 수 있습니다.
  - 요청 취소(`AbortController`), 요청 순번/버전 비교, "마지막 의도만 반영" 같은 장치를 고려하세요.
- **대량 데이터 성능**:
  - 5,000개를 그대로 DOM에 그리면 느립니다. **화면에 보이는 것만 렌더**하는 전략(가상화) 또는 그에 준하는 방법을 고민하세요.
  - 불필요한 리렌더를 줄이는 것(메모이제이션, 안정적인 `key`)도 중요합니다.
- **상태 구조**: "서버에서 온 데이터"와 "낙관적으로 바꾼 임시 상태"를 어디서 어떻게 관리할지 먼저 설계하면 나머지가 쉬워집니다.
- **테스트**: `src/lib/tasks.test.ts`에 순수 함수 테스트 예시가 있습니다. 같은 방식으로 핵심 로직을 검증하세요.

컴포넌트 구조, 상태 관리 방식, 폴더 구성 등은 **자유롭게 재구성**해도 됩니다.

## 6. 제공물 & mock API

| 구분 | 내용 |
|------|------|
| UI | 3컬럼 칸반 + 카드 렌더 + HTML5 드래그 앤 드롭(기본) |
| mock API | `src/mocks/` — 지연·실패·409를 흉내내는 MSW 핸들러 |
| 시드 | 태스크 **5,000개** (결정적 생성, 모두 동일) |
| 세팅 | Vite + React 18 + TypeScript(strict) + Vitest |

### mock API 명세

모든 요청에 **200~800ms 랜덤 지연**이 있습니다.

| Method | Endpoint | 동작 | 실패 |
|--------|----------|------|------|
| GET | `/api/tasks` | 전체(5,000개) 반환 | 드물게 500 |
| POST | `/api/tasks` | 생성 | ~15% 500 |
| PATCH | `/api/tasks/:id` | 부분 수정 | ~15% 500, `version` 불일치 시 **409**(+서버 최신 상태) |
| DELETE | `/api/tasks/:id` | 삭제 | ~15% 500 |

- **낙관적 동시성 제어**: 각 태스크에 `version`(정수)이 있습니다. `PATCH`는 알고 있는 `version`을 함께 보내야 하며, 서버 값과 다르면 **409 Conflict**와 함께 **서버의 최신 상태**를 응답합니다. 성공 시 `version`이 1 증가합니다.
- **실패율·지연은 `src/mocks/config.ts`에서 조절**합니다. 개발 중 `WRITE_FAILURE_RATE = 1`로 두면 모든 쓰기가 실패해 **롤백 동작을 확인**하기 좋습니다. **제출 전에는 기본값으로 되돌려 주세요.**
- ⚠️ **mock API를 우회해 localStorage로 대체하는 것은 금지**입니다(비동기 처리가 과제의 본질). 단, 다중 탭 동기화 등 부가 목적의 localStorage 사용은 무방합니다.

## 7. 실행 방법

```bash
npm install      # postinstall 에서 public/mockServiceWorker.js 자동 생성
npm run dev      # 개발 서버
npm test         # 유닛 테스트 (Vitest)
npm run build    # 타입체크 + 프로덕션 빌드
```

Node 18+ 권장.

### GitHub Pages 배포

Vite `base`를 저장소 이름으로 지정해 빌드하세요. service worker 등록 경로가 자동으로 따라갑니다.

```bash
# 예) 저장소 이름이 my-repo 라면
VITE_BASE=/my-repo/ npm run build
```

> 💡 Windows Git Bash에서는 `/my-repo/`의 앞 슬래시가 경로로 변환될 수 있으니, PowerShell(`$env:VITE_BASE="/my-repo/"; npm run build`)이나 `MSYS_NO_PATHCONV=1`를 사용하세요.

배포 방법 두 가지:
- **GitHub Actions**: `deploy.yml.example`을 `.github/workflows/deploy.yml`로 옮기고 push → Settings → Pages → Source: **GitHub Actions** (workflow 스코프 토큰 필요)
- **브랜치 배포**: 로컬 빌드 후 `dist`를 `gh-pages` 브랜치로 push → Settings → Pages → Source: **Deploy from a branch → gh-pages**

## 9. 평가 기준 (상세)

총점 100점, 4개 영역으로 평가합니다. **개수보다 완성도**, **결과보다 근거**를 봅니다.

### ① 견고함 — 비동기 정합성·성능·엣지케이스 (40%)
가장 큰 비중입니다. 채점 시 **실패를 강제로 재현**해가며 확인합니다.
- 저장이 실패해도 UI가 **정확히 롤백**되고 사용자에게 알림이 가는가
- 카드를 빠르게 연속 이동해도 **최종 상태가 서버와 일치**하는가 (경쟁 상태)
- 5,000개에서 검색/필터/드래그가 **부드러운가**
- 로딩/에러(재시도)/빈 상태가 모든 지점에서 처리되는가
- (가점) 다중 탭 동기화, 재시도 큐, 409 충돌 처리 등

### ② 코드 품질 — 구조·재사용성·테스트 (30%)
- 컴포넌트/모듈 책임 분리, 재사용성 (거대 컴포넌트·과도한 prop drilling 지양)
- **서버 상태와 클라이언트(낙관적) 상태**를 구분해 다루는 상태 설계
- 핵심 순수 로직에 대한 **유닛 테스트의 존재와 정확성** (테스트가 없으면 이 항목 감점)
- TypeScript를 안전하게 활용(strict 유지, `any` 남발 지양)
- 가독성·일관성, 정리된 커밋 히스토리

### ③ 결정의 근거 — DECISIONS.md / AI_USAGE.md (20%)
- 롤백·상태구조·성능 트레이드오프를 **왜 그렇게 했는지** 서술했는가
- "일부러 명세하지 않은 것"(오프라인/409/재시도)에 대한 **본인의 결정과 근거**가 있는가
- "A를 위해 B를 포기했다" 같은 **트레이드오프 인지**가 드러나는가
- AI 답을 **검증·거부·수정한 구체적 사례**가 있는가

### ④ 기본 사용성 (10%)
- 직관적이고 막힘 없는 조작 흐름
- 확인 다이얼로그·피드백(토스트 등) 기본 UX
- (다크모드·화려한 애니메이션은 채점하지 않음)

> **면접 라이브 세션**: 과제는 통과 필터이며, 최종 판정은 면접에서 이루어집니다. 본인의 제출 코드를 열고 약 30분간 **새 기능을 함께 확장**합니다(AI 사용 가능). 자기 코드를 이해하고 그 위에서 확장·설명할 수 있어야 합니다.

## FAQ

**Q. mock API 대신 localStorage를 써도 되나요?**
A. 안 됩니다. 비동기 처리가 과제의 본질입니다. (부가 목적의 localStorage는 무방)

**Q. 모든 요구사항을 다 구현해야 하나요?**
A. 아니요. **P1을 견고하게** 끝내는 것이 P2·P3를 얕게 여러 개 하는 것보다 높은 평가를 받습니다.

**Q. 드래그/UI 라이브러리를 써도 되나요?**
A. 네. `DECISIONS.md`에 선택 이유만 간단히 남겨 주세요.

**Q. 성능 기준이 애매합니다.**
