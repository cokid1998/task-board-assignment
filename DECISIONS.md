# 설계 결정 (DECISIONS)

> 이 문서는 평가에서 **20%**를 차지합니다. "무엇을 했는지"보다 **"왜 그렇게 했는지"**를 봅니다.
> 아래 항목을 채워 주세요. 코드로 답이 되는 부분은 파일 경로를 함께 적어도 좋습니다.

## 1. 상태 구조

- 서버 상태와 클라이언트(낙관적) 상태를 어떻게 나눴나요? 어디서 관리하나요?
  - 나누지 않았습니다. `Board.tsx의 tasks`하나로 관리합니다.
- 이렇게 잡은 이유와, 고려했다가 버린 대안은?
  - 서버, 클라이언트 상태로 나누면 두 상태를 동기화 시키는 코드가 추가되며 리소스가 들어갈꺼라고 생각해서 굳이 둘로 나누지 않았습니다.
  - React-Query 도입 고려: 캐싱, 재시도, 낙관적 업데이트를 강력하게 제공하지만, 라이브러리의 동작 방식에 종속되어 세부 로직을 완전히 커스텀하기 어려울 수 있다고 판단해 도입하지 않고 직접 구현했습니다.
  - `tasks`를 `App`으로 옮기기: 헤더에서 생성 모달을 열 때 `Board`내부 로직에 `App`이 접근하지 못하는 문제가 있었습니다. `tasks`전체를 `App`으로 옮기는 것도 고려했으나 코드 수정범위가 커져서 "추가"버튼을 `Column`내부로 옮겨 해결했습니다.

## 2. 낙관적 업데이트 & 롤백

- 이동/수정/삭제/생성 시 UI를 먼저 반영하고 실패하면 어떻게 되돌리나요?
  - 네 가지 작업 모두 동일한 흐름을 따릅니다. 서버 요청 전에 tasks를 먼저 낙관적으로 변경해 화면에 즉시 반영하고, 그 직전 상태(원본)를 별도로 보관해둡니다. 서버 요청이 성공하면 응답으로 받은 값으로 tasks를 다시 확정하고, 실패하면 보관해둔 원본으로 tasks를 되돌리고 사용자에게 alert로 알립니다.
- 롤백을 "서버 재조회" vs "이전 스냅샷 복원" 중 무엇으로 했고 그 이유는?
  - **이전 스냅샷 복원** 방식을 택했습니다. 서버 재조회는 실패했다는 사실 하나를 확인하기 위해 5,000개 전체를 다시 받아와야 해 네트워크 낭비가 크고, 재조회 자체도 실패할 수 있어 오히려 복잡도가 늘어난다고 판단했습니다.

  - 반면 스냅샷 복원은 이미 로컬에 갖고 있는 값으로 즉시 되돌릴 수 있어 구현이 간단했습니다.

## 3. 경쟁 상태(race condition)

- 같은 카드를 빠르게 연속 이동할 때 최종 상태가 어긋나지 않도록 무엇을 했나요?
  - 요청마다 고유 순번(mutationId)을 발급하고, 카드별로 최신 순번을 추적해 판별했습니다.

  - 이동 요청마다 전역카운터로 고유한 `mutationId`를 발급하고, 카드 id를 key로 하는 Map에 `{ mutationId, original }`을 저장합니다.

  - 응답이 오면 Map에 기록된 그 카드의 최신 `mutationId`와 자신의 `mutationId`가 같은지 확인합니다. 다르면 요청을 보낸 후 추가 이동이 있었다는 뜻이므로 무시합니다.

  - 409를 받으면 서버 최신 버전으로 재요청해, 사용자의 마지막 의도를 서버에도 반영합니다.
  - 순번 비교를 택한 이유: 응답 순서는 통제할 수 없지만 "카드별로 가장 나중 시도가 무엇인지"는 로컬에서 정확히 알 수 있기 때문입니다. AbortController도 고려했으나, 서버에 이미 요청한걸 취소시키는게 아니라 프론트에서 연결을 끊기만 하기 때문에 근본적인 해결법이 아니라고 생각했습니다.

## 4. 대량 데이터 성능 (5,000개)

- 검색/필터/드래그를 부드럽게 만들기 위해 무엇을 선택했고 **무엇을 포기**했나요?
  - `React.memo`: `Card`를 감싸 `task`가 안 바뀐 카드는 리렌더를 건너뜁니다. 검색 결과에 남은 카드는 재계산되지 않습니다.
  - `useTransition`: 검색 입력을 화면 표시용(inputValue)과 필터링용(query)으로 분리, query 갱신을 낮은 우선순위로 처리해 타이핑이 끊기지 않게 했습니다.

- 가능하면 측정 근거(예: 최적화 전/후 체감·프로파일)를 남겨 주세요.
  - 별도 프로파일링 도구 없이 브라우저에서 직접 확인했습니다. `memo` 적용 전에는 검색할 때마다 남아있는 카드까지 전부 재계산되어 입력이 끊겼고, 적용 후에는 사라지는 카드만 제거되고 나머지는 재계산이 생략되어 지연이 사라졌습니다.

## 5. 정답이 없던 결정들 (명세서에 명시되지 않은 항목)

- **네트워크 완전 단절** 시 동작을 어떻게 정의했나요? 왜?
  - 네트워크 단절만을 위한 에러 처리는 하지 않고 500 에러와 동일하게 처리했습니다. 네트워크 단절 시 처리는 생각해본 적이 없어서 단순히 500 에러로 처리했는데, 다시 생각해보면 토스트나 모달로 네트워크 단절 안내 정도로 처리하는 게 더 나은 방법이었을 것 같습니다.

- **409 충돌** 시 사용자에게 무엇을 보여주고 어떻게 해소하나요? 왜?
  - 아무것도 보여주지 않고, 서버가 알려준 최신 버전으로 조용히 재요청했습니다.
  - 409에러가 발생할 때마다 사용자에게 알림을 준다는건 사용자 행동을 끊기게 해서 UX적으로 좋지 못하다고 판단했고, 이미 지나간 시도에 대한 알림을 받을 수 있어 혼란을 줄 수 있다고 판단했습니다.

- 실패한 쓰기의 **재시도**: 자동 vs 수동 — 무엇을 택했고 이유는?
  - 수동을 선택했습니다. 쓰기 요청이 실패하면 UI를 롤백하고 alert만 띄우고, 재요청은 별도로 하지 않습니다.

  - 자동 재시도는 몇 번까지, 얼마나 간격을 두고 시도할지 등 로직의 복잡성이 크다고 생각해서 굳이 구현하지 않았습니다. UI를 롤백해서 사용자가 실패한 동작을 반복하는 것만으로 충분히 재시도를 대체할 수 있다고 판단했습니다.

## 6. 그 외 트레이드오프 / 남은 한계

- 시간 제약으로 포기한 것, 알고 있는 버그·한계, 더 있었다면 했을 것.
  - 스크롤 가상화 미구현: 처음 접하는 개념이라 시간이 많이 걸릴 것 같아 구현하지 못했지만, 시간이 있다면 꼭 구현해보고 싶은 기능입니다.
  - `Board` 컴포넌트의 핸들러 함수들(`handleDeleteTask` 등)을 `useReducer`로 리팩토링하고 싶었는데, 시간 관계상 하지 못한 게 아쉽습니다.
