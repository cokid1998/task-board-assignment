# AI 도구 활용 (AI_USAGE)

> AI 사용은 **권장**됩니다. 이 문서는 "얼마나 썼는지"가 아니라
> **AI가 준 답을 어떻게 검증·판단했는지**를 봅니다. 솔직하게 적어 주세요.

## 1. 사용한 도구

- **Claude**

## 3. AI 답을 그대로 쓰지 않고 검증·거부·수정한 사례 (중요)

- 어떤 답이 틀렸거나 부족했고, 어떻게 알아챘으며, 무엇으로 고쳤나요?

1. 이중 try/catch 구조를 거부:

   AI가 409 재요청 처리를 위해 이중 try/catch를 포함한 코드를 제시했으나, 구조가 복잡해지는 것에 비해 발생 빈도가 낮은 엣지케이스라고 판단해 "그냥 try/catch 하나만 쓰는 방식으로 가자"고 요청했습니다.

2. React.memo 무력화 원인을 직접 조사해서 AI의 첫 진단을 보완:

   CRUD기능을 완성시켜보니 이전에 구현해놨던 input태그 타이핑 최적화가 적용이 안됐었습니다. AI에게 말했더니 useTransition과 관련된 해결법을 제시했지만 소용이 없었고 커밋 히스토리를 직접 되짚어 "삭제 기능을 추가한 시점부터 버벅임이 시작됐다"는 사실을 알아낸 뒤 AI에게 전달했고, 그제서야 Column에서 onDelete={() => onDelete(t.id)} 형태로 매 렌더마다 새 함수가 생성되어 React.memo가 무력화된다는 정확한 원인을 함께 찾아 구조를 수정했습니다.

## 4. 스스로 판단해서 결정한 부분

- React Query 미도입: 캐싱/재시도/낙관적 업데이트를 표준 패턴으로 제공한다는 장점이 있었지만, 라이브러리 동작 방식에 종속되어 세부 로직 커스텀이 제한될 수 있다고 판단해 직접 구현했습니다.

- CRUD에는 race condition 방어(mutationId)를 적용하지 않음: 모든 CRUD에 동일하게 적용하는 방향도 검토했으나, 삭제/생성/수정은 이동만큼 연속 발생 가능성이 낮다고 판단해 적용하지 않았습니다.

- 409 재요청의 재요청 실패는 처리하지 않음: 이중 try/catch로 재요청 실패까지 처리하는 방법도 검토했으나, 발생 빈도가 낮은 엣지케이스라 판단해 의도적으로 배제했습니다.
